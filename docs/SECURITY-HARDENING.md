# Security Hardening — sjsunsin.org

This document captures the threat model and step-by-step runbook for keeping
`sjsunsin.org` hard to attack. It is the source of truth for what needs to be
configured **outside** the code repository (Cloudflare, GitHub repo settings,
DNS, the registrar).

## Threat model in plain English

`sjsunsin.org` is an Astro **static site** deployed to **GitHub Pages**. There
is no backend, no database, no user input that the server processes. That
fundamentally limits the realistic threats:

| #  | Threat                                       | Where it lives                    | What stops it                                          |
| -- | -------------------------------------------- | --------------------------------- | ------------------------------------------------------ |
| 1  | Cross-site scripting (XSS)                   | Browser, in our markup            | CSP in `astro.config.mjs`; no `set:html` in source     |
| 2  | Clickjacking                                 | Browser, our pages in iframes     | `frame-ancestors 'none'` + `X-Frame-Options: DENY`     |
| 3  | MIME sniffing                                | Browser                           | `X-Content-Type-Options: nosniff`                      |
| 4  | Referrer leak to third parties               | Browser                           | `Referrer-Policy: strict-origin-when-cross-origin`     |
| 5  | Mixed-content downgrade                      | Browser                           | HSTS preload + `upgrade-insecure-requests` in CSP      |
| 6  | npm supply chain                             | CI build                          | Dependabot + `npm audit` step + CodeQL                 |
| 7  | GitHub Action compromise                     | CI                                | Workflow `permissions:` minimized; `persist-credentials: false` on checkout |
| 8  | Domain / subdomain takeover                  | DNS / registrar                   | Registrar lock + 2FA + DNSSEC + CAA records            |
| 9  | Third-party CDN poisoning (Google Fonts)     | Browser                           | `font-src` allowlist + `crossorigin="anonymous"`       |
| 10 | Volumetric DDoS, vuln-scanner noise          | Edge                              | Cloudflare proxy (free plan is sufficient)             |
| 11 | Phishing using look-alike subdomain          | DNS                               | Don't create extra subdomains; monitor cert transparency |
| 12 | Accidental secret commit                     | Repo                              | GitHub secret scanning + push protection enabled       |

The Cloudflare "1519 threats blocked" email is item **#10** — almost all of it
is automated scanner noise looking for `wp-admin`, `xmlrpc.php`, `.env`, etc.
For a static site, **none of those probes can succeed** even without
Cloudflare in front. The email is an upsell, not an incident.

## What the code already does

- **CSP** is generated by Astro 6 from `security.csp` in `astro.config.mjs`
  and emitted as a `<meta http-equiv>` tag on every page. Inline `<script>`
  and `<style>` produced by Astro are hashed automatically.
- **Referrer-Policy**, **color-scheme**, and `robots` meta tags are set in
  `src/layouts/BaseLayout.astro`.
- All `target="_blank"` links carry `rel="noopener noreferrer"`.
- Google Fonts CSS is requested with `crossorigin="anonymous"` and
  `referrerpolicy="no-referrer"`.
- `public/_headers` ships the full header set for **Cloudflare Pages /
  Netlify** if the site is ever migrated. **GitHub Pages ignores this file.**
- `public/robots.txt`, `public/security.txt`, and
  `public/.well-known/security.txt` are present.
- `src/pages/404.astro` ships a custom 404 page.
- `src/pages/security-policy.astro` mirrors `SECURITY.md` for users who hit
  the public link.
- GitHub workflows are scoped to least-privilege `permissions:` blocks and
  `persist-credentials: false` on checkout.
- Dependabot watches **npm + github-actions** weekly.
- CodeQL (security-extended + security-and-quality), OSSF Scorecard, and
  Dependency Review run on every push / PR.

## What still requires manual configuration

GitHub Pages does **not** let you set HTTP response headers. To get HSTS,
`X-Frame-Options`, `X-Content-Type-Options`, `Permissions-Policy`, and a
real (header-based) CSP, you need to put Cloudflare in front as a
**proxy** (orange cloud), not just for DNS.

### 1. Cloudflare DNS + proxy

In `dns-import.txt` today the records resolve straight to GitHub Pages
(`185.199.110.153 / 185.199.111.153`). Switch the proxy on for the apex and
`www`:

```
sjsunsin.org.        A     185.199.108.153   PROXIED
sjsunsin.org.        A     185.199.109.153   PROXIED
sjsunsin.org.        A     185.199.110.153   PROXIED
sjsunsin.org.        A     185.199.111.153   PROXIED
www.sjsunsin.org.    CNAME sjsunsin.org.     PROXIED
```

Once proxied, do this in the Cloudflare dashboard:

1. **SSL/TLS → Overview** → set to **Full (strict)**.
2. **SSL/TLS → Edge Certificates**:
   - Always Use HTTPS: **On**
   - HTTP Strict Transport Security: **On**, `max-age=63072000`,
     `includeSubDomains`, `preload`
   - Minimum TLS Version: **TLS 1.2** (1.3 also enabled)
   - Opportunistic Encryption: **On**
   - TLS 1.3: **On**
   - Automatic HTTPS Rewrites: **On**
3. **Speed → Optimization → Early Hints**: **On** (safe with our CSP).
4. **Security → Settings → Bot Fight Mode**: **On**.
5. **Security → Settings → Security Level**: **Medium** is fine for a static
   site. Raise to **High** if scanner noise increases.
6. **Security → WAF → Managed rules**: enable Cloudflare Managed Ruleset +
   Cloudflare OWASP Core Ruleset (paranoia level 1).
7. **Network → Onion Routing**: **On** (lets Tor users reach the site without
   getting flagged).

### 2. Cloudflare Transform Rules — response headers

These rewrite/add headers the same way `_headers` would on Cloudflare Pages.
Cloudflare dashboard → **Rules → Transform Rules → Modify Response Header**.
Create one rule, "All requests", with the actions below (use **Set** for each):

| Header                          | Value |
| ------------------------------- | ----- |
| `Strict-Transport-Security`     | `max-age=63072000; includeSubDomains; preload` |
| `X-Content-Type-Options`        | `nosniff` |
| `X-Frame-Options`               | `DENY` |
| `Referrer-Policy`               | `strict-origin-when-cross-origin` |
| `Cross-Origin-Opener-Policy`    | `same-origin` |
| `Cross-Origin-Resource-Policy`  | `same-site` |
| `X-Permitted-Cross-Domain-Policies` | `none` |
| `Origin-Agent-Cluster`          | `?1` |
| `Permissions-Policy`            | `accelerometer=(), camera=(), geolocation=(), gyroscope=(), magnetometer=(), microphone=(), payment=(), usb=(), interest-cohort=()` |

Note: do **not** set a `Content-Security-Policy` HTTP header at the
Cloudflare edge unless you also remove the `<meta>` CSP that Astro emits.
Browsers AND-combine the two and you will get a stricter policy than either
one alone, which often blocks legitimate assets. Pick one; the Astro-emitted
meta-CSP is preferred because it carries the per-page script/style hashes.

### 3. Cloudflare Page Rules / Configuration Rules — caching

Cloudflare dashboard → **Rules → Configuration Rules**:

- URL matches `*sjsunsin.org/_astro/*` → Browser Cache TTL: 1 year, Edge
  Cache TTL: 1 month, Cache Level: Cache Everything.
- URL matches `*sjsunsin.org/images/*` or `*sjsunsin.org/videos/*` → Browser
  Cache TTL: 1 month.
- URL matches `*sjsunsin.org/.well-known/*` → Cache Level: Bypass (so
  `security.txt` updates land immediately).

### 4. Cloudflare WAF — custom rules

In **Security → WAF → Custom rules**, create:

1. **Block obvious scanner paths.** Expression:
   ```
   (http.request.uri.path eq "/wp-login.php") or
   (http.request.uri.path eq "/xmlrpc.php") or
   (http.request.uri.path eq "/.env") or
   (http.request.uri.path matches "^/wp-admin($|/)") or
   (http.request.uri.path matches "^/(phpmyadmin|adminer|pma)($|/)") or
   (http.request.uri.path matches "\\.(php|asp|aspx|cgi|jsp)$")
   ```
   Action: **Block**. Reduces logged "threat" volume substantially.

2. **Block known bad ASNs and TOR exit nodes from POST.** Expression:
   ```
   (http.request.method eq "POST") and (not http.request.uri.path matches "^/$")
   ```
   Action: **Managed Challenge**. A static site has no legitimate POST
   targets at all.

3. **Rate-limit aggressive crawlers.** Expression:
   ```
   (http.user_agent contains "MJ12bot") or
   (http.user_agent contains "DotBot") or
   (http.user_agent contains "SemrushBot") or
   (http.user_agent contains "PetalBot")
   ```
   Action: **Managed Challenge**, with **Rate limit** = 10 req/min/IP.

### 5. CAA records (issue authority binding)

In Cloudflare DNS, add the following CAA records to lock CA issuance to the
ones GitHub Pages and Cloudflare actually use:

```
sjsunsin.org.  CAA  0 issue "letsencrypt.org"
sjsunsin.org.  CAA  0 issue "digicert.com"
sjsunsin.org.  CAA  0 issue "pki.goog"
sjsunsin.org.  CAA  0 issuewild ";"
sjsunsin.org.  CAA  0 iodef "mailto:security@sjsunsin.org"
```

`issuewild ";"` disallows any wildcard certs; we don't need them.

### 6. DNSSEC

In Cloudflare → **DNS → Settings → DNSSEC**: enable it and copy the DS
record to the registrar. Without DNSSEC, a domain hijack via DNS forgery is
possible upstream of Cloudflare.

### 7. Registrar hardening

At the domain registrar (the one that owns `sjsunsin.org`):

- Enable **registrar lock** (clientTransferProhibited, clientUpdateProhibited).
- Enable **2FA** on the registrar account, ideally with a hardware key.
- Use a dedicated email for the registrar account that is **not** the same
  email used for the GitHub repo.
- Set the DS record from step 6.

### 8. GitHub repo settings (one-time)

Repository → **Settings**:

- **General → Default branch**: `main`. Disable force pushes.
- **General → Pull Requests**: enable "Always suggest updating pull request
  branches" and "Automatically delete head branches".
- **Branches → Add classic branch protection / ruleset** for `main`:
  - Require a pull request before merging
  - Require status checks: `CodeQL`, `Dependency Review`, the `audit` job
    from `deploy.yml`
  - Require linear history
  - Require signed commits (recommended)
  - Restrict force pushes
  - Restrict deletions
- **Code security → Secret scanning**: **On**, push protection: **On**.
- **Code security → Dependabot alerts**: **On**.
- **Code security → Dependabot security updates**: **On**.
- **Code security → Code scanning**: rely on the CodeQL workflow.
- **Code security → Private vulnerability reporting**: **On**.
- **Actions → General**:
  - Allow only "Actions created by GitHub" + "verified creators" + an
    explicit allowlist of action SHAs.
  - Workflow permissions: **Read repository contents and packages**.
  - Fork pull request workflows from outside collaborators: **Require
    approval for first-time contributors who are new to GitHub**.
- **Pages**: enforce HTTPS (it should already be on; check after enabling
  Cloudflare proxy).

### 9. Collaborator hygiene

- Audit `Settings → Collaborators` quarterly.
- Require 2FA on every collaborator account.
- Use SSO if/when the org moves to a paid GitHub plan.
- Remove any GitHub App that you don't recognize under
  `Settings → Integrations → Installed GitHub Apps`.

### 10. Email auth (DMARC / SPF / DKIM)

`security@sjsunsin.org` lives somewhere — wherever that is, set:

```
sjsunsin.org.            TXT  "v=spf1 -all"
_dmarc.sjsunsin.org.     TXT  "v=DMARC1; p=reject; rua=mailto:dmarc@sjsunsin.org; sp=reject; aspf=s; adkim=s"
```

If the domain doesn't send mail at all, the `-all` SPF + `p=reject` DMARC
prevents look-alike phishing claiming to be from `@sjsunsin.org`.

## Verification

After applying the steps above, verify:

1. Browse `https://sjsunsin.org/` and check the response headers in
   DevTools → Network. You should see HSTS, X-Frame-Options, X-Content-Type-Options,
   Referrer-Policy, COOP, CORP, Permissions-Policy, and a CSP `<meta>`.
2. Run `https://securityheaders.com/?q=sjsunsin.org`. Target grade: **A**.
3. Run `https://observatory.mozilla.org/analyze/sjsunsin.org`. Target: **A+**.
4. Run `https://www.ssllabs.com/ssltest/analyze.html?d=sjsunsin.org`.
   Target: **A+**, no weak ciphers, HSTS preload eligible.
5. Run `https://hstspreload.org/?domain=sjsunsin.org` and submit to the
   preload list once `max-age=63072000; includeSubDomains; preload` is live.
6. Run `https://internet.nl/site/sjsunsin.org/` for an end-to-end report.
7. View `https://sjsunsin.org/.well-known/security.txt` in a browser.
8. Confirm Dependabot, CodeQL, and Scorecard runs are green in the **Actions**
   tab.

## Operational practices

- Treat any push that touches `.github/workflows/`, `package.json`,
  `package-lock.json`, or `astro.config.mjs` as **security-sensitive**.
  Require review.
- When upgrading Astro across a major version, re-verify CSP — Astro's
  hashing of inline scripts/styles can change with the version.
- When adding any third-party script/iframe (analytics, embeds), update the
  `directives` in `astro.config.mjs` first; ship in the same PR as the embed.
- If you ever switch hosts, the `public/_headers` file already has the right
  set for Cloudflare Pages / Netlify — verify nothing was renamed.
- Keep `Expires:` in `security.txt` no more than 12 months in the future;
  refresh annually.
